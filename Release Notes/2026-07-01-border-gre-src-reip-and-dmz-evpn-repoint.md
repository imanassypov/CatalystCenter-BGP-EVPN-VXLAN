# Release Note — 2026-07-01

## Border GRE Source Loopback Re-IP, DMZ EVPN Neighbor Repoint, and Spine Prefix-List Correction

### Summary

During live commissioning of the border-to-DMZ GRE/OSPF 100 overlay (introduced in
`2026-06-25-border-dmz-evpn-gre-overlay.md`), the GRE tunnel from Border-02 to the DMZ
was found stuck in OSPF `INIT` state. Root-cause analysis via the core NX-OS routing tables
revealed an IP address conflict: Border-02's `Loopback1` (GRE tunnel source, `198.19.1.8`)
was identical to Core-01's `Loopback0`. Consequently, GRE proto-47 packets destined for
`198.19.1.8` were absorbed by Core-01 rather than forwarded to Border-02, producing a
one-way adjacency. The fix was to move both border GRE source loopbacks (`Loopback1`) into
the `10.100.2.0/29` range (alongside the existing DMZ hub address `10.100.2.1`), eliminating
the conflict and providing a logically cohesive GRE source address block.

Separately, the spine `MCLUSTER-LOOPBACKS` prefix-list was corrected: the legacy
`198.19.1.200/32` (DMZ EVPN identity) entry was removed because in the GRE/OSPF 100
design that address is reachable via OSPF 100 through the tunnel itself — injecting it
into OSPF 1 from the spine would decouple peer reachability from tunnel state and prevent
`fall-over route-map NODE-LOOPBACKS` from triggering on tunnel failure.

---

### Root Cause

| Symptom | Cause |
|---|---|
| Border-02 Tunnel10 — 0 packets input, OSPF 100 neighbor INIT | GRE dst `198.19.1.8` resolved to Core-01 Lo0 (attached/local) — packets never forwarded to border |
| Border-01 Tunnel10 — OSPF FULL, packets flowing | `198.19.1.7` (Border-01 Lo1) had no conflict; correctly forwarded via core |
| Verified via Core-01: `198.19.1.8/32 ubest/mbest: 2/0, attached` | Core-01 owns `198.19.1.8` on its own Loopback0 |

### Changes Made

#### 1. Border-01 and Border-02 — Loopback1 Re-IP (live, manual)

| Device | Old Loopback1 IP | New Loopback1 IP |
|---|---|---|
| Border-01 | `198.19.1.7/32` | `10.100.2.5/32` |
| Border-02 | `198.19.1.8/32` | `10.100.2.6/32` |

Commands applied:
```
! Border-01
interface Loopback1
 ip address 10.100.2.5 255.255.255.255

! Border-02
interface Loopback1
 ip address 10.100.2.6 255.255.255.255
```

Tunnel source references the interface (`tunnel source Loopback1`) — no tunnel command
change required. The IOS-XE tunnel line-state re-evaluated automatically.

#### 2. DMZ — Tunnel Destinations Updated (live, manual)

The DMZ Tunnel10/Tunnel11 destinations pointed to the old border Loopback1 IPs.

| Tunnel | Old Destination | New Destination |
|---|---|---|
| Tunnel10 (→ Border-01) | `198.19.1.7` | `10.100.2.5` |
| Tunnel11 (→ Border-02) | `198.19.1.8` | `10.100.2.6` |

Commands applied on dmz1:
```
interface Tunnel10
 tunnel destination 10.100.2.5
interface Tunnel11
 tunnel destination 10.100.2.6
```

#### 3. Spine01 and Spine02 — FABRIC-LOOPBACKS Prefix-List Updated (live, manual)

Seq 7 and seq 8 replaced to track the new border Loopback1 IPs:

```
! Before (both spines)
ip prefix-list FABRIC-LOOPBACKS seq 7 permit 198.19.1.7/32
ip prefix-list FABRIC-LOOPBACKS seq 8 permit 198.19.1.8/32

! After (both spines)
ip prefix-list FABRIC-LOOPBACKS seq 7 permit 10.100.2.5/32
ip prefix-list FABRIC-LOOPBACKS seq 8 permit 10.100.2.6/32
```

`FABRIC-LOOPBACKS` feeds `fall-over route-map FABRIC-LOOPBACKS` on spine iBGP
peer-sessions — tracking the border Loopback1 ensures iBGP fast-fallover responds
correctly if a border GRE source goes down.

Additionally, the legacy aggregate entry was removed from both spines (pre-existed from
CML lab startup config; never generated by templates; overly permissive):
```
no ip prefix-list FABRIC-LOOPBACKS seq 10 permit 198.19.1.0/24 ge 32
```

#### 4. Spine01 and Spine02 — MCLUSTER-LOOPBACKS Prefix-List Corrected (live, manual)

The `198.19.1.200/32` (DMZ EVPN identity, Loopback0) entry was removed. It was a legacy
artifact from the pre-GRE design where borders peered EVPN directly via OSPF 1 reachability.
In the GRE/OSPF 100 design, `198.19.1.200` is reachable exclusively via OSPF 100 through the
GRE tunnel — this is intentional and provides correct failover coupling.

The `10.100.2.1/32` (DMZ Loopback1, GRE hub source) entry was retained. This is the only
address that borders need a specific OSPF 1 route to, so that GRE encapsulation does not
rely on default-route ECMP (which can hash via a core with asymmetric reachability).

```
! Before (both spines)
ip prefix-list MCLUSTER-LOOPBACKS seq 15 permit 198.19.1.200/32   ← removed
ip prefix-list MCLUSTER-LOOPBACKS seq 20 permit 10.100.2.1/32

! After (both spines)
ip prefix-list MCLUSTER-LOOPBACKS seq 20 permit 10.100.2.1/32
```

#### 5. DMZ — BGP EVPN Neighbors Repointed (live, manual — Phase 2)

The DMZ BGP EVPN peers were still pointing to the pre-change border `Loopback0` addresses.
Repointed to border `Loopback2` (OSPF 100 RID / EVPN overlay identity) to match the
template-governed border side.

```
router bgp 65003
 ! Remove legacy Loopback0 peers
 no neighbor 198.19.1.5
 no neighbor 198.19.1.6

 ! Add Loopback2 peers
 neighbor 10.101.1.2 inherit peer-session OVERLAY-Border-EVPN-PEER-SESSION-POLICY
 neighbor 10.101.2.2 inherit peer-session OVERLAY-Border-EVPN-PEER-SESSION-POLICY

 address-family ipv4
  no neighbor 10.101.1.2 activate
  no neighbor 10.101.2.2 activate

 address-family l2vpn evpn
  neighbor 10.101.1.2 activate
  neighbor 10.101.1.2 send-community both
  neighbor 10.101.1.2 inherit peer-policy OVERLAY-Border-EVPN-PEER-POLICY
  neighbor 10.101.1.2 route-map EVPN-TYPE5-ONLY in
  neighbor 10.101.1.2 route-map DMZ-policy-out out
  neighbor 10.101.2.2 activate
  neighbor 10.101.2.2 send-community both
  neighbor 10.101.2.2 inherit peer-policy OVERLAY-Border-EVPN-PEER-POLICY
  neighbor 10.101.2.2 route-map EVPN-TYPE5-ONLY in
  neighbor 10.101.2.2 route-map DMZ-policy-out out
```

The DMZ peer-session `OVERLAY-Border-EVPN-PEER-SESSION-POLICY` (remote-as 65001,
update-source Loopback0 `198.19.1.200`, ebgp-multihop 255) was left unchanged — correct
for the new design.


2. `DEFN_LOOP_MCLUSTER` extended with `gre_src` key for template-driven prefix-list
   generation:


`gre_src` = DMZ GRE hub source (redistributed into OSPF 1 on spines via MCLUSTER-LOOPBACKS).

#### `FABRIC-EVPN.j2`

New block added after `route-map FABRIC-LOOPBACKS`, spine-only, gated on `DEFN_TUNNELS`:

```jinja
{% if DEVICE_HOSTNAME in DEFN_NODE_ROLES['RR'] and DEFN_TUNNELS is defined and DEFN_TUNNELS | length > 0 %}
{% for remote in DEFN_NODE_ROLES['MCLUSTER'] %}
ip prefix-list MCLUSTER-LOOPBACKS seq {{loop.index * 10 + 5}} permit {{DEFN_LOOP_MCLUSTER[remote]['gre_src']}}/32
{% endfor %}
!
{% endif %}
```

Rendered output (current single DMZ remote, loop.index=1):
```
ip prefix-list MCLUSTER-LOOPBACKS seq 15 permit 10.100.2.1/32
```

The `ip` field (`198.19.1.200`) is deliberately excluded from this prefix-list — see
"Design Rationale" below.

`route-map BGP-TO-OSPF permit 30 / match ip address prefix-list MCLUSTER-LOOPBACKS`
already exists on both spines from the CML baseline. No template change needed for the
route-map itself.

---

### Design Rationale — Why `198.19.1.200` is Excluded from OSPF 1

| Scenario | With `198.19.1.200` in OSPF 1 | Without (correct) |
|---|---|---|
| Tunnel healthy | Both OSPF 1 + OSPF 100 routes present; BGP peer up | OSPF 100 route only; BGP peer up |
| Tunnel fails | OSPF 100 route gone, but OSPF 1 still has `198.19.1.200` → `fall-over` does NOT trigger | OSPF 100 route gone, no OSPF 1 fallback → `fall-over` triggers → BGP session torn down correctly |
| Result if left in | BGP session stays up, traffic blackholes silently | BGP session torn down; control plane reflects data-plane failure |

`198.19.1.200` must be reachable **only** via OSPF 100 (i.e., through the GRE tunnel)
so that peer reachability and tunnel state remain coupled. The DMZ already redistributes
its `Loopback0` into OSPF 100 via `route-map LOOPBACK0-INTO-OSPF100`.

### Operational Impact

- GRE/OSPF 100 overlay between borders and DMZ is now fully operational on both paths.
- BGP EVPN sessions between borders and DMZ are Established and exchanging prefixes.
- `fall-over route-map NODE-LOOPBACKS` on borders will correctly tear down the EVPN session
  if the GRE tunnel fails (DMZ EVPN identity reachable only via OSPF 100).
- No impact to leaf, spine EVPN, or L3OUT/multicast functionality.
- Next CatC provisioning run will be idempotent: template-generated entries match live config.
